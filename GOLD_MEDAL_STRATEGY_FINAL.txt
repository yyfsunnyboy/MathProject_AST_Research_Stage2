【3】Domain 分類與設計思路
════════════════════════════════════════════════════════════════════════════════════════════════════

為什麼要分 Domain？
────────────────
✓ 不同數學領域需要不同的「核心助手函數」
  例：calculus 需要多項式運算，但 complex_numbers 需要複數格式化
  
✓ 減少 Token 成本：MASTER_SPEC 只包含該 domain 的相關邏輯
  例：整數運算 domain 的 MASTER_SPEC 不需要向量運算代碼
  
✓ 提高代碼生成準確度：Qwen 14B 面對簡化的 MASTER_SPEC 更容易專注

建議的 Domain 劃分：
────────────────

1. **calculus** (微積分)
   ├─ gh_ApplicationsOfDerivatives
   ├─ gh_AreaBetweenCurves
   ├─ gh_AverageValueOfContinuousFunction
   └─ gh_DefiniteIntegralAndArea

2. **geometry_and_conics** (幾何與圓錐曲線)
   ├─ gh_DistancesRelatedToLines
   ├─ gh_GeometricApplicationsOfEllipse
   ├─ gh_JudgingTheRelationshipOfCircleAndLine
   └─ gh_EquationOfALine

3. **linear_algebra** (線性代數)
   ├─ gh_VectorDotProduct
   ├─ gh_MatrixMeaningAndEquality
   └─ gh_CrossProductOfSpaceVectors

4. **trigonometry** (三角函數)
   ├─ gh_BasicTrigonometricIdentities
   └─ gh_SineAndCosineFunctionGraphs

5. **complex_and_polar** (複數與極坐標)
   ├─ gh_PolarFormOfComplexNumbers
   └─ gh_ConversionBetweenRectangularAndPolarCoordinates

6. **probability_statistics** (機率統計)
   ├─ gh_ConditionalProbability
   └─ gh_BayesTheorem

7. **sequences_series** (數列級數)
   └─ gh_Sequence

【4】技術實現路線圖
════════════════════════════════════════════════════════════════════════════════════════════════════

Phase 1: 數據庫與模型擴展 (本週)
────────────────────────────────
□ 在 models.py 中添加 domain 和 domain_confidence 字段
□ 為 20 支技能標注 domain
□ 為每個技能分配優先度 (calculus=1, geometry=2, 等等)

Phase 2: Domain-Specific MASTER_SPEC (下週)
────────────────────────────────────────────
□ 創建 7 個 domain-specific MASTER_SPEC 模板
  ├─ 每個模板 500-800 字符（而不是 2500+ 混合模板）
  ├─ 包含該 domain 特有的：
  │  ├─ 核心數學概念
  │  ├─ 常用輔助函數清單
  │  ├─ 變數命名規範
  │  └─ 常見邊界情況
  └─ 保留統一的「Healer 修復邏輯」層
  
□ 修改 prompt_architect.py：
  ├─ detect_domain(skill_id) 函數
  ├─ select_master_spec_template(domain) 函數
  └─ 將 MASTER_SPEC + textbook_examples 結合

Phase 3: 統一 Healer 層設計 (第三週)
──────────────────────────────────
□ Healer 保持單一、不分 domain（因為修復邏輯通用）
□ 在 Healer 中添加 domain-aware checks：
  ├─ 如果 domain == 'calculus': 檢查多項式運算
  ├─ 如果 domain == 'complex_numbers': 檢查複數格式
  └─ 如果 domain == 'geometry': 檢查距離/角度計算

Phase 4: 快速驗證與調整 (第四週)
────────────────────────────────
□ 測試 Qwen 14B 在每個 domain-specific MASTER_SPEC 下的表現
□ 對比性測試：
  ├─ 單一巨大 MASTER_SPEC vs 7 個小 MASTER_SPEC
  ├─ 測試指標：成功率、Healer 修復率、生成速度
  └─ 預期結果：小 MASTER_SPEC 效果更好且更快

Phase 5: Ablation Study (第五週)
────────────────────────────────
□ 在 20 支精選技能上執行 Ablation Study
  ├─ Ab1: BARE_MINIMAL + domain-specific MASTER_SPEC (NO Healer)
  ├─ Ab2: domain-specific MASTER_SPEC (NO Healer)
  ├─ Ab3: domain-specific MASTER_SPEC + 統一 Healer
  └─ 蒐集 10 × 20 = 200 個資料點

【6】預期風險與對策
════════════════════════════════════════════════════════════════════════════════════════════════════

風險 1: 某個 domain 的 MASTER_SPEC 太難設計
對策: 參考 Gemini 代碼中的「輔助函數」來設計 MASTER_SPEC 模板

風險 2: 20 個技能太多，無法在期限內完成
對策: 先實施「calculus + geometry」(8個技能)，再擴展到其他 domain

風險 3: Healer 對某些 domain 的代碼效果差（如複數運算）
對策: 在 Healer 中為該 domain 添加特殊規則，或調整生成 Prompt

風險 4: Qwen 14B 無法達到 Gemini 水準
對策: 這本身就是 Ablation Study 要驗證的（失敗也是結果）
