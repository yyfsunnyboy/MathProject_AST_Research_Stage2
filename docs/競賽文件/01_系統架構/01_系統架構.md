## 1. 系統整體架構

### 1.1 技術堆疊

```
前端層 (UI/UX)
├─ HTML5 + CSS3 + JavaScript
├─ Jinja2 模板引擎
└─ 互動式 Dashboard

應用層 (Business Logic)
├─ Flask 2.0+ Web 框架
├─ Flask-SQLAlchemy ORM
├─ Flask-Login 認證系統
└─ REST API 設計

核心業務層 (AI 生成)
├─ Architect (設計規格生成)
├─ Code Generator (代碼生成)
├─ Healer (統一代碼清理/修復)
└─ Evaluator (MCRI 評分系統)

資料持久層
├─ SQLite (開發)
└─ PostgreSQL (生產)
```

## 3. 實驗管理系統

### 3.1 實驗架構概覽

```
準備階段 (Setup)
├─ 匯出 Golden Prompts (一次性)
│  ├─ experiments/golden_prompts/{skill}_Ab1.txt
│  ├─ experiments/golden_prompts/{skill}_Ab2.txt
│  └─ experiments/golden_prompts/{skill}_Ab3.txt
└─ 設置實驗目錄結構

實驗階段 (Execution)
├─ 執行批次實驗 (900 次總計)
│  ├─ 20 個技能 × 3 模型 × 3 策略 × 5 樣本
│  └─ 自動儲存到分類目錄
└─ 自動生成評測報告 (JSON)

分析階段 (Analysis)
├─ 統計 Token 成本、修復次數等
├─ 計算 MCRI 評分
└─ 生成對比報告
```

## 4. 統一代碼清理 Healer

### 4.1 設計理念

**問題背景**:
傳統修復方案會導致反復修改，例如：
- 先刪除污染變數 (如 `IntegerOps = 0`)
- 後發現未定義，再次添加
- 結果代碼仍然有污染 ❌

**解決方案**:
一次性 AST 遍歷，完成所有修復 → **避免衝突**

### 4.2 統一 Healer 架構

```
UnifiedCleanupHealer
│
├─ 【第一遍掃描】ProblemScanner
│  ├─ 收集所有本地定義 (class/function)
│  ├─ 掃描所有賦值語句
│  ├─ 標記重複定義
│  ├─ 標記污染賦值 (預定義名稱)
│  └─ 標記變量順序問題
│
└─ 【第二遍清理】ProblemFixerß
   ├─ 刪除所有重複定義 (一次)
   ├─ 刪除所有污染賦值 (一次) ← 不再反復
   └─ 修復變量順序
```
