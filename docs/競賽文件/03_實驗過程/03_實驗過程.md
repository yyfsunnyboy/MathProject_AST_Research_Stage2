## 1. 實驗方法論與設計

### 1.1 核心目標

設計一個嚴謹的實驗框架，科學地驗證以下兩個獨立假設：
- **H1**: Prompt Engineering 能顯著提升生成內容品質 (L2/L4 分數)
- **H2**: 自我修復 (Healer) 機制能提升執行穩定性與強健性 (L1/L3 分數)

### 1.2 實驗矩陣設計：3×3 Full Factorial

| 維度 | 選項 | 基於原則 |
|:---|:---|:---|
| **Prompt 工程層級** | Ab1 (Bare) / Ab2 (Engineered) / Ab3 (Full) | 鑑別 Prompt 的邊際效應 |
| **樣本複雜度** | S1 (Simple) / S2 (Complex) / S3 (Abstract) | 檢驗通用性與適應性 |
| **模型規模** | 7B/14B (Local Qwen) / Cloud (Gemini) | 成本效益分析 |

### 1.3 控制變因與依變項

#### 獨立變因 (Independent Variables)

**1. Prompt 工程層級**:
- **Ab1 (Bare Prompt)**: 極簡指令，無工具庫，無詳細規格
  ```plaintext
  【角色】K12 數學演算法工程師
  【任務】實作 generate(level=1) 函數，生成「導數的應用」題目
  【輸出格式】{'question_text': str, 'correct_answer': str, 'answer': str, 'mode': 1}
  ```
  
- **Ab2 (Engineered Prompt)**: 完整 MASTER_SPEC + 工具 Stubs + 格式規範
  - 詳細的技術要求
  - 代碼安全模式 (禁止無限迴圈、標準化答案格式等)
  - Stub Injection 工具介面

- **Ab3 (Full System)**: Ab2 Prompt + 自動修復機制 (Healer)
  - 前端：同 Ab2 的 Engineered Prompt
  - 後端：9 層 Healer Pipeline (正則→AST→深度修復)
  - 核心變因分離：Ab2 vs Ab3 = 唯一差異是 Healer

**2. 樣本複雜度**:
- **S1 (Simple)**: 整數運算，無分數/根號
- **S2 (Complex)**: 分數/根號/多項式
- **S3 (Abstract)**: 函數/微積分/抽象代數

#### 依變項 (Dependent Variables)

- **MCRI 總分** (0-100分)
- **各層級分數** (L1, L2, L3, L4)
- **成功率** (Pass Rate)
- **Token 效率** (Token/成功案例)

### 1.4 科學性原則

1. **變因分離** (Variance Isolation)
   - Ab1 vs Ab2 vs Ab3：每次只改變一個要素
   - Ab2 和 Ab3 使用完全相同的 Prompt（唯一差異：Healer 的開關）

2. **重複性** (Reproducibility)
   - 固定的 Golden Prompts（不再動態生成）
   - 完整的相機條件記錄
   - 可重現的實驗環境

3. **公平性** (Fairness)
   - 所有 Ablation 都執行「統一基礎清理」
   - 相同的沙箱環境與評測標準

## 2. MCRI 評測系統

### 2.1 系統概述

**MCRI** (Modular Code Reliability Index) V4.2.2 是針對 AI 生成的教學內容所設計的綜合評測框架。傳統 `Pass@k` 指標只關注「程式會不會崩潰」，無法衡量「題目好不好」。MCRI 從 4 個層級評估代碼質量。

### 2.2 評分結構與權重 (總分: 100)

| 層級 | 評估對象 | 權重 | 核心問題 |
|:---|:---|---:|:---|
| **L1** | Execution Integrity | 20% | 程式能在沙箱中編譯並執行嗎？ |
| **L2** | Technical Compliance | 20% | 輸出格式是否符合 Schema？ |
| **L3** | Robustness & Fairness | 30% | 題目經得起各種學生的「不規範輸入」嗎？ |
| **L4** | Pedagogical Quality | 30% | 這個題目適合教學嗎？(數學美感) |

### 2.3 各層級詳解

#### **L1: Execution Integrity (20%)**

測試點：
- 代碼是否能在 Python 沙箱中編譯？
- 是否在 5 秒內完成執行（超時檢查）？
- 是否會拋出未捕獲的異常？

評分規則：
- ✅ 通過全部測試：20分
- ⚠️ 超時或異常：5-10分
- ❌ 失敗：0分

#### **L2: Technical Compliance (20%)**

測試點：
- 回傳格式是否符合 Schema？
  ```python
  {'question_text': str, 'correct_answer': str, 'answer': str, 'mode': int}
  ```
- 所有必需鍵值都存在嗎？
- 數據類型是否正確？
- 變數命名是否遵循規範？

評分規則：
- ✅ 完全符合：20分
- ⚠️ 缺少可選欄位：15分
- ❌ 缺少必需欄位：0分

#### **L3: Robustness & Fairness (30%)**

**L3.1 Internal Consistency (子評 15%)**
- 核心問題：`check(correct_answer)` 必須回傳 True 嗎？
- 測試：定義的 `correct_answer` 是否真的是正確答案？
- 評分：完全符合 15分，部分符合 7-8分，不符合 0分

**L3.2 External Robustness (子評 15%*)
- 核心問題：當學生輸入各種變體時，判分會不會出錯？*(這是最難的一關)*
- 例子：同一個答案 `2/3`，學生可能輸入：
  - `2/3`（標準形式）
  - `0.666...`（小數形式）
  - `2 / 3`（帶空格）
  - `.666666`（無整數部分）
- 評分規則：完全符合 15分，90%符合 12分，70%符合 8分，<50%符合 0分

#### **L4: Pedagogical Quality (30%)**

**L4.1 Numeracy (子評 15%)**
- 核心問題：數字是否「漂亮」？(避免教學上的噪聲)
- 檢查項：
  - 分母是否 > 1000？(扣 -5分)
  - 小數是否超過 4 位？(扣 -3分)
  - 是否存在零係數項？如 `0x + 3`（扣 -10分）
  - 是否存在隱藏的係數？如 `1x` 而非 `x`（扣 -3分）
- 評分：滿分 15分，逐項扣分

**L4.2 Visuals (子評 15%)**
- 核心問題：數學表達是否使用規範的 LaTeX？
- 檢查項：
  - 分數是否用 `\frac{a}{b}` 而非 `a/b`？
  - 指數是否用 `^` 而非 `**`？
  - 是否正確轉義特殊符號？
- 評分：滿分 15分，逐項扣分

### 2.4 案例分析

#### 例子：多項式導數問題

題目：求 $f(x) = 3x^2 - 2x + 1$ 的導數

| 評估維度 | 代碼示例 | 評分 | 說明 |
|:---|:---|---:|:---|
| **L1** | `def generate(): return {...}` | 20 | 執行成功且無超時 |
| **L2** | 正確格式 `{'question_text': '...', ...}` | 20 | Schema 完全符合 |
| **L3.1** | `check('6x - 2')` = True | 15 | 答案內容正確 |
| **L3.2** | `check('6*x - 2')` = True | 15 | 開放性檢查通過 |
| **L4.1** | 答案為 `\frac{6x-2}{1}` 而非簡化 | -5 | 分母不應為 1 |
| **L4.2** | 使用 `\frac` 代替 `/` | 15 | LaTeX 格式完整 |
| **總分** | - | **80** | 良好質量 |

## 3. Ablation 模式詳解

### 3.1 三種 Ablation 小組的完整定義

#### **Ab1: Bare Prompt (基準組)**

**設計思想**: 模擬「天真的用戶」，只給出最基本的自然語言指令

Prompt 內容：
```plaintext
【角色】
你是 K12 教育領域的數學演算法工程師。

【任務】
實作 `def generate(level=1, **kwargs)` 函數，生成「導數的應用」相關的數學作業題目。

【輸出格式】
函數必須返回一個字典，包含以下鍵值對：
- question_text (str): 題目描述，可包含 LaTeX
- correct_answer (str): 正確答案
- answer (str): 答案說明
- mode (int): 難度級別 (1~3)

【要求】
1. 題目應基於多項式函數及其導數
2. 答案必須是數學上正確的
3. 生成的題目應該有難度變異性

---

**特點**:
- ✅ 簡潔、易理解
- ❌ 無工具輔助、無具體規範
- ❌ 完全依賴模型的「常識」

**預期效果**:
- 只有高能力模型能理解並正確執行
- 低能力模型容易產生如下錯誤：
  - 邏輯錯誤 (正確答案不正確)
  - 格式錯誤 (輸出不符合 Schema)
  - 語法錯誤 (代碼無法編譯)

#### **Ab2: Engineered Prompt (工程化組)**

**設計思想**: 提供完整的技術規格書，讓模型「按圖施工」

Prompt 內容 (來自 MASTER_SPEC 與 Stub Injection 系統)：

```plaintext
【角色】
K12 教算法工程師

【工具庫】(Stubs Only)
class PolyCalc:
    @staticmethod
    def differentiate_poly(coeffs: list[float]) -> list[float]:
        """求多項式導數。
        Args: coeffs - [a0, a1, a2, ...] 表示 a0 + a1*x + a2*x^2 + ...
        Returns: 導數的係數列表
        """
        pass
        
    @staticmethod
    def poly_to_latex(coeffs: list[float]) -> str:
        """將多項式係數轉為 LaTeX 字符串
        例: [3, -2, 1] -> "3 - 2x + x^2"
        """
        pass

【數據流程】
1. 使用 PolyCalc.differentiate_poly() 計算導數
2. 使用 PolyCalc.poly_to_latex() 將結果轉為 LaTeX
3. 返回格式化的題目

【安全模式】
- ✅ 只使用提供的工具函數
- ❌ 禁止 eval() 或 exec()
- ❌ 禁止無限迴圈 (while True)
- ❌ 禁止自己實作 PolyCalc (將會被自動刪除)

【格式規範】
- 所有數學式必須用 $ 包裹
- 答案必須是「純結果」格式
- 例: 正確 ✅ "3x - 2" 
      錯誤 ❌ "答案是 3x - 2"
      錯誤 ❌ "3x - 2 = 0"

【代碼質量指標】
- L1: 必須在 5 秒內完成執行
- L2: 輸出格式必須符合 Schema
- L3: `check(correct_answer)` 必須回傳 True
- L4: 答案必須符合數學規範
```

**特點**:
- ✅ 詳細指導、具體工具、明確規範
- ✅ 即使是 7B 小模型也能理解
- ⚠️ 系統複雜度增加

**預期效果**:
- 成功率大幅提升 (60-70%)
- **Prompt 層級衝突** 改善但仍可能出現

#### **Ab3: Full System (完整系統組)**

**設計思想**: 在 Ab2 基礎上，加入自動修復機制

構成：
- **前端 Prompt**: 與 Ab2 完全相同 (變因分離的關鍵)
- **後端 Healer**: 9 層自動修復管道

Healer 管道層次：
1. **L0 清理**: 移除 Markdown 標記、多餘空行
2. **L1 Regex 修復**: 括號匹配、縮進錯誤
3. **L2 AST 解析**: 語法樹檢查
4. **L3 Duplicate 清理**: 移除重複定義的類別
5. **L4 Loop Breaker**: 檢測並修復無限迴圈
6. **L5 Import Check**: 驗證所有導入
7. **L6 Safe Eval**: 嘗試安全執行
8. **L7 Deep AST**: 深層邏輯修復
9. **L8 Context Aware**: 基於 MCRI 反饋的修復

**特點**:
- ✅ 極高的通過率 (98%+)
- ✅ 代碼品質最高
- ❌ 複雜度最大、維護成本高

**實驗結果**:
- Ab1 Pass Rate: ~15%
- Ab2 Pass Rate: ~65%
- Ab3 Pass Rate: ~98%

### 3.2 Ablation 組間的差異分析表

| 要素 | Ab1 | Ab2 | Ab3 | 差異說明 |
|:---|:---:|:---:|:---:|:---|
| Prompt 長度 | ~200 chars | ~2000 chars | ~2000 chars | Ab2/3 相同 |
| 工具提供 | ❌ | ✅ (Stub) | ✅ (Stub) | Ab2/3 相同 |
| 代碼範本 | ❌ | ✅ | ✅ | Ab2/3 相同 |
| Healer 修復 | ❌ | ❌ | ✅ | **唯一區別** |
| **預期成功率** | **15%** | **65%** | **98%** | 遞增 |
| 單位成本 ($/1000 tokens) | $0.001 | $0.05 | $0.05 | Ab2=Ab3 成本相同 |

## 4. 技術突破與里程碑

### 4.1 里程碑時間線

#### 📅 2026-01-15: MCRI 評測系統確立 (里程碑 1)

**核心問題**: 如何科學地評估 AI 生成的教學內容？

**突破**: 提出 **MCRI V4.2 架構**，超越傳統「Pass@k」指標

**意義**:
- L1/L2：關注工程基礎（執行、格式）
- L3：關注健壯性（學生的各種輸入變體）
- L4：關注教學價值（數學美感）

**心得**:
> 「教育軟體的重點不是代碼寫得多漂亮，而是學生用起來順不順手。如果答案是 3.1415926535... 而不是 π，那這個題目就是失敗的。」

---

#### 📅 2026-01-29: Healer 的誕生 (里程碑 2)

**核心問題**: AI 生成的代碼 5-10% 會出現語法錯誤或無限迴圈，傳統 Retry 效率低

**突破**: 開發 **9 層 Healer Pipeline**
- 從簡單 Regex 修復到強大 AST 修復
- 引入 `Safe Eval` 與 `Loop Breaker` 防止惡意和死循環
- 創新式：**源頭修復勝於事後修復**

**技術創新**:
- **Regex 層** (L1-L2): 快速修復括號、縮進
- **AST 層** (L3-L5): 深層次語法檢查與修復
- **Context 層** (L6-L8): 基於 MCRI 反饋的智能修復

**心得**:
> 「Regex 只能修皮毛，AST 才能修筋骨。當我們看到 Healer 自動修好了那些缺括號、縮進錯誤的代碼時，真的覺得這個系統『活』過來了。」

**實際數據**:
- 修復成功率: 92%
- 平均修復時間: 200ms per error
- 邊際成本: 幾乎可忽略不計

**修復方案**:
1. 確保 `ablation_id` 正確傳遞至 `code_generator.py`
2. 添加多層驗證檢查
3. 每次生成都驗證 Ab1/Ab2/Ab3 的 Prompt 確實不同

**教訓**: 建立了嚴格的團隊驗證流程

#### 📅 2026-02-15：Stub Injection 技術突破 (里程碑 5)

**核心問題**: 
- 將完整工具代碼放進 Prompt → Token 爆炸 (60% 浪費)
- 不給工具 → AI 亂寫 (幻覺)

**突破解決方案**: **Stub Injection (樁代碼注入)**

流程：
1. **Prompt 階段**: 只給 AI 看工具的「皮」 (Stubs: 介面定義 + Docstring)
   ```python
   class PolyCalc:
       @staticmethod
       def differentiate_poly(coeffs: list[float]) -> list[float]:
           """求多項式導數。..."""
           pass  # AI 不可實作
   ```

2. **生成階段**: AI 呼叫這些工具
   ```python
   result = PolyCalc.differentiate_poly([1, 2, 3])
   ```

3. **執行階段**: `code_generator.py` 強制注入工具的「骨肉」 (Full Implementation)
   ```python
   # 系統自動注入完整實現
   class PolyCalc:
       @staticmethod
       def differentiate_poly(coeffs):
           return [i * coeffs[i] for i in range(1, len(coeffs))]
   ```

4. **安全清理**: 使用 AST 解析器，強制移除 AI 自己實作的 Stub，確保系統工具是唯一真理

**效果**:
| 指標 | 改善 |

```python
def _build_prompt(
    skill_id: str,
    ablation_id: int,
    db_master_spec: str,
    use_golden_prompt: bool = False  # 新參數
) -> tuple:
    """
    構建 Prompt，支援動態生成或 Golden Prompt 讀取
    
    Args:
        use_golden_prompt: 若為 True，從檔案讀取；否則動態生成
    
    Returns:
        (prompt_text, prompt_length, metadata)
    """
    
    if use_golden_prompt:
        # 方式 1：讀取 Golden Prompt
        try:
            golden_file = f"experiments/golden_prompts/temp/{skill_id}_Ab{ablation_id}.txt"
            # 註：Ab2/Ab3 使用相同檔案
            if ablation_id == 3:
                golden_file = f"experiments/golden_prompts/temp/{skill_id}_Ab2.txt"
            
            with open(golden_file, 'r', encoding='utf-8') as f:
                prompt = f.read()
            return (prompt, len(prompt.split()), {"source": "golden_prompt", "file": golden_file})
        except FileNotFoundError:
            # Fallback：動態生成
            print(f"⚠️  Golden Prompt 檔案未找到，降級為動態生成")
            use_golden_prompt = False
    
    if not use_golden_prompt:
        # 方式 2：動態生成 (原有邏輯)
        prompt = PromptBuilder.build(
            master_spec=db_master_spec,
            ablation_id=ablation_id,
            textbook_example=get_textbook_example(skill_id),
            topic=get_topic(skill_id),
            skill_id=skill_id
        )
        return (prompt, len(prompt.split()), {"source": "dynamic_build"})
```

## 6. 關鍵實驗發現

### 6.1 發現 A：Prompt 層級衝突 (The Hierarchy Conflict)

#### 現象
在 Ab2 實驗中發現一個反直覺的現象：**過度詳細的 Prompt 指令反而導致模型困惑**。

#### 具體案例
當要求：
```plaintext
【指令 1】只能使用 IntegerOps 工具箱
【指令 2】生成分數題目 (分子/分母格式)
```

模型陷入困境：
- 選項 A：違反指令 1，使用其他工具（LLM 不願意違反明確指令）
- 選項 B：遵守指令 1，無法生成分數（邏輯矛盾）
- **結果**: 生成代碼崩潰或無限迴圈

#### 根本原因
MASTER_SPEC (高層指令) 與 Tool Constraints (底層指令) 之間的**邏輯衝突**

#### 解決方案
引入 **Stub Injection 技術**，讓模型只需關注「接口調用」，隱藏實作細節，消除了上下文中的衝突資訊

#### 量化影響
- 衝突狀態：Ab2 成功率 50-55%，間歇性 Timeout
- 修復後：Ab2 成功率 65-75%，穩定性大幅提升

### 6.2 發現 B：Healer 的必要性

#### 數據對比

| 階段 | Pass Rate | 描述 |
|:---|---:|:---|
| **Ab1 (Bare Prompt)** | ~15% | 大量語法錯誤、邏輯錯誤 |
| **Ab2 (Engineered Prompt)** | ~65% | 邏輯正確但偶爾格式錯誤 |
| **Ab3 (Engineered + Healer)** | ~98% | Healer 修復了絕大多數格式問題 |

#### 質量分解

| 評估層級 | 改善來源 | 累積效果 |
|:---|:---|---:|
| L1 (執行完整性) | Ab1→Ab2：+50% | 50% |
| L2 (技術規範) | Ab1→Ab2：+40% | 90% |
| L3 (強健性) | Ab2→Ab3：+33% | 98% |
| L4 (教學質量) | Ab2→Ab3：+15% | 100% |

#### 結論

## 10. 歷史歸檔與技術演進

### 10.1 早期技術突破

#### 2026-02-08：Healer V2.8 與重複定義之戰

**事件**: 發現 AI 生成的代碼中存在重複的 `class IntegerOps` 定義

**問題**:
- 系統注入的 `class IntegerOps` (完整版)
- AI 自己寫的 `class IntegerOps` (殘廢版)
- 結果：`Redefinition Error` 或 `AttributeError`

**解決**: 實作了 `remove_duplicate_class_definitions()`（Regex層級）

**意義**: 首次認識到「Tool Injection」與「AI Generation」之間的衝突，開啟了與 AI 幻覺對抗的長期戰爭

#### 2026-02-04：MCRI V4.4 - 追求數學之美

**升級點**:
- L4.3 (Quality Control)：新增扣分邏輯
  - 零係數項 -10分（如 `0x + 3`）
  - 符號未簡化 -3分（如 `(-2)` 而非 `-2`）
- L5 (Complexity)：SymPy 數學複雜度分析

**意義**: 從「讓程式能跑」轉向「讓題目好用」

#### 2026-02-01：Loop Breaker V2.0 - AST 的勝利

**背景**: 早期 Regex 暴力替換 (`while True` → `for _ in range`) 經常破壞縮排，導致 `IndentationError`

**升級**: 引入 AST 模組進行正確的迴圈修復與縮排調整

**意義**: 證明單靠 Regex 無法處理 Python 複雜語法，直接啟發了後來 Stub Injection 的 AST 清理機制

### 10.2 重要教訓

#### 教訓 1：驗證的重要性

> 「Verify, Verify, Verify!」
> 
> Ablation Bug 讓我們意識到，沒有驗證的實驗結果都是可疑的。從此建立了多層驗證體系。

#### 教訓 2：簡設計勝於複雜設計

> 「Less is More」
>
> Ab1 雖然簡陋，但其簡潔性常常被高估。而 Ab3 的複雜性需要正視，但不是壞事——因為它帶來了更好的教學品質。

#### 教訓 3：源頭修復勝於事後修復

> 「在源頭解決問題比在下游 healing 要划算得多。」
>
> Stub Injection 就是這個哲學的體現：解決 Prompt 層級衝突後，Healer 的工作量減少，效率提升。
